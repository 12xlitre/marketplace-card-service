# Discovery: Marketplace Card Service

## Общее понимание

Сервис задуман как визуальный кабинет для команды, которая ведет магазины клиентов на маркетплейсах и массово работает с карточками товаров.

Главная ценность первой версии:
- удобство работы команды;
- повышение качества обработки карточек;
- работа с большими объемами карточек;
- ускорение ручных процессов;
- контроль изменений и статусов.

Деньги, биллинг и коммерческие метрики не являются приоритетом для MVP.

## Роли

В кабинете должны работать:
- менеджер проекта;
- технический специалист, который также может выполнять SEO- и контент-задачи;
- руководитель.

Предполагается, что менеджер ведет проект и согласует действия, технический специалист выполняет аудит и подготовку изменений, руководитель контролирует процесс и результат.

## Первый маркетплейс

Первым поддерживаем Wildberries.

## Объект работы

Карточка товара в широком смысле:
- название;
- описание;
- ключи;
- фото;
- характеристики;
- цены;
- остатки;
- категории.

Для первой версии приоритет:
- аудит карточек;
- предложения по улучшениям;
- объяснение качества текущего состояния: что плохо/хорошо и почему;
- изменения названий/заголовков;
- изменения характеристик;
- работа с ключами.

## Целевой сценарий

Желаемый путь:

1. Команда получает доступ к магазину Wildberries/API с достаточными правами или получает конкретный список карточек для работы.
2. Сервис проводит аудит магазина и карточек.
3. Сервис находит возможные улучшения.
4. Сервис показывает предложения и аргументы: что исправить, почему это важно, какой вариант лучше.
5. Специалист проверяет и корректирует предложения; менеджер видит процесс, но не применяет изменения.
6. Технический специалист нажимает действие на массовое применение после ручной проверки.
7. Сервис вносит изменения в карточки.
8. Команда видит статусы, ошибки и итоговый результат.

## Режимы доступа и охват работы

Сервис должен учитывать два рабочих сценария:

1. Полный доступ к магазину.
   Команда видит весь кабинет продавца, может загрузить все карточки, провести общий аудит, отфильтровать проблемные позиции и готовить массовые правки по большому объему.

2. Работа по выбранным карточкам.
   Клиент может передать не весь магазин, а конкретные карточки, артикулы, nmID, SKU или ссылку/таблицу с позициями. В этом случае сервис должен создавать ограниченную рабочую область только по выбранному набору карточек.

Для интерфейса важно показывать охват работы на уровне кабинета селлера:
- полный магазин;
- выбранные карточки;
- количество карточек в текущем охвате;
- источник набора: API, выгрузка, таблица, ручной список;
- возможность расширить или заменить набор карточек.

Для логики важно, чтобы аудит, фильтры, согласование, массовые правки, история и ошибки работали не только по магазину целиком, но и по конкретному выбранному набору карточек.

## Объемы

Сервис должен быть полезен как для больших партий, например 1000 карточек, так и для небольших задач, например 30 карточек.

Причина появления идеи: ручная работа с большим объемом карточек становится неудобной и плохо масштабируется.

## Фактический процесс Wildberries

Есть два стартовых сценария.

### Магазина Wildberries еще нет

1. Создать магазин/кабинет продавца.
2. Запросить у продавца товарную матрицу.
3. Подобрать семантическое ядро.
4. Провести оптимизацию карточек:
   - создать заголовки с нуля;
   - заполнить характеристики;
   - создать описания;
   - загрузить фотографии;
   - рассчитать цену и маржинальность;
   - установить цены;
   - создать склад.
5. Для FBS:
   - создать склад;
   - загрузить остатки.
6. Для FBO:
   - создать поставку;
   - загрузить данные по поставке;
   - оформить поставку.

### У клиента уже есть кабинет Wildberries

1. Получить доступы к кабинету.
2. Проанализировать матрицу товаров.
3. Дальше процесс почти такой же, как при создании магазина с нуля: семантика, заголовки, характеристики, описания, фото, цены, склады, остатки/поставки.

### Текущие способы работы

- В личном кабинете Wildberries вручную выполняются почти все действия.
- Часть действий выполняется через Excel/CSV-таблицы. По текущему опыту через таблицы можно сделать практически все операции с карточками, но это не одна универсальная таблица.
- API пока не используется. API-ключ может сгенерировать только владелец кабинета.
- Повторяющиеся действия появляются примерно раз в месяц или по запросу клиента.
- На тип товара, категорию или клиента зависят заголовки, характеристики, описания и расчет маржинальности.

### Где теряется время и возникают ошибки

Больше всего времени уходит на:
- характеристики;
- заголовки;
- описания;
- расчет маржинальности.

Чаще всего ошибки возникают в:
- характеристиках;
- ценах.

Первый кандидат на автоматизацию:
- создание и варианты заголовков;
- подбор семантического ядра для каждой подкатегории;
- заполнение и проверка характеристик;
- создание описаний.

## Доступы, данные и источники

Для начала работы нужен доступ в кабинет администратора. Обычно он выдается через номер телефона, к которому привязан личный кабинет Wildberries.

Для API нужен ключ, который генерирует владелец кабинета. На этапе discovery предполагается, что нужны права владельца, но это нужно уточнить по реальным методам WB API: сервису не стоит требовать больше прав, чем необходимо для загрузки, аудита и обновления карточек.

Перед аудитом сейчас используются:
- MPStats для анализа конкурентов, запросов в карточках и позиций;
- выгрузки данных о карточках для будущей корректировки;
- Excel и CSV как основные форматы выгрузок.

Обязательные поля в выгрузке:
- артикул;
- наименование/заголовок;
- характеристики, разные для каждой категории;
- цена;
- описание;
- фото или ссылка на фото.

На текущем опыте все нужные данные можно получить выгрузками, без обязательного ручного просмотра. Корректность загрузки должна проверяться через раздел/механизм ошибок: Wildberries показывает, какие данные заполнены некорректно.

## Аудит карточки

Провести аудит карточки означает проверить:
- запросы в карточке;
- характеристики;
- заголовок;
- описание;
- наличие и релевантность запросов;
- цену в сравнении с конкурентами;
- пустые поля;
- корректность заполненных данных;
- смысловую точность данных;
- количество и качество фото.

Критерии оценки названия, описания, характеристик, ключей и категории:
- сравнение с конкурентами;
- релевантные запросы;
- данные MPStats;
- опыт специалиста и экспертная оценка.

Фото проверяются уже на первом этапе.

Готового чек-листа хорошей карточки пока нет. Правила отличаются по категориям товаров, поэтому сервису нужен слой правил по категории/подкатегории и механизм обновления этих правил.

Оценка качества карточки должна быть не только числовой, но и объяснимой:
- плохо, потому что поле пустое, нерелевантное, слабее конкурентов или содержит ошибку;
- нормально, потому что критичных ошибок нет, но есть потенциал улучшения;
- хорошо, потому что карточка заполнена, релевантна запросам и конкурентоспособна по содержанию.

## Предложения по улучшениям

Новое название карточки формируется на основе:
- конкурентов;
- запросов;
- данных MPStats;
- высокочастотных запросов;
- запросов топовых конкурентов в категории;
- целевых запросов, подходящих именно этому товару.

Важное ограничение Wildberries: наименование до 60 символов с пробелами.

Ключи подбираются по спросу и релевантности товару. Источник SEO-данных и частотности на текущем процессе - MPStats.

Характеристики нужно добавлять или исправлять на основе:
- конкурентов;
- запросов;
- MPStats;
- правил конкретной категории.

Важное продуктовое правило: сервис не должен применять предложения "почти без риска". Все предложения требуют ручной проверки.

Желательный формат предложения:
- что плохо сейчас;
- почему это плохо;
- что предлагается;
- почему предлагается именно так;
- 2-3 варианта исправления, если это возможно.

Опасность не в наличии предложений, а в нерелевантности. Если система будет анализировать поток карточек и выдавать релевантные варианты, готовые к обработке специалистом, это должно заметно ускорить работу.

## Массовое обновление и контроль риска

Массово потенциально можно обновлять почти все поля карточек, но это нужно раскладывать по конкретным API-методам и таблицам Wildberries.

Рискованные зоны:
- цены - самый высокий риск;
- обязательные атрибуты;
- характеристики с ограничением количества значений в одной ячейке;
- атрибуты, правила которых зависят от категории.

Сейчас массовое обновление происходит через:
- личный кабинет;
- импорт Excel;
- ручную работу с таблицами.

API пока не используется.

Ограничения Wildberries, которые нужно учитывать:
- технические работы;
- задержки модерации;
- обязательные поля;
- категорийные ограничения характеристик;
- ограничения количества значений атрибута;
- ошибки формата или значения атрибута.

Типовые ошибки при обновлении:
- неверно заполненный атрибут;
- пустое обязательное поле, например цена.

Если часть карточек обновилась, а часть нет, сервис должен явно показать:
- какие карточки не обновились;
- какое поле не обновилось;
- причину или текст ошибки;
- что делать дальше.

Перед изменениями нужно хранить старую версию карточки. Нужен backup, история и возможность отката. После массового обновления сервис должен сделать чек-ап: перечитать данные, сравнить ожидаемое и фактическое состояние, показать успешные изменения и расхождения.

## Согласование

Согласование желательно делать внутри сервиса.

Идея кабинетов:
- кабинет менеджера;
- переход из кабинета менеджера в личный кабинет/рабочую область магазина;
- рабочее пространство технического специалиста для аудита, правок и публикации.

Менеджер не применяет изменения в карточки. Технический специалист может видеть все данные и применять изменения. Менеджер видит почти всю рабочую информацию, но не технические ошибки и не может вносить изменения без технического специалиста.

Финальное утверждение зависит от проекта:
- иногда изменения применяются без отдельного согласования с клиентом;
- иногда требуется согласование клиента;
- если продавец будет присутствовать в системе, нужно предусмотреть отдельный сценарий клиентского согласования;
- если продавца в системе нет, технический специалист принимает финальное решение.

Лучше утверждать каждую карточку отдельно. Для больших объемов, например больше 50 строк, интерфейс должен показывать карточки страницами/партиями, а не бесконечной таблицей.

Перед нажатием "применить" технический специалист должен проверить все данные. Изменения цен и обязательных атрибутов должны требовать дополнительного подтверждения.

Нужно хранить:
- комментарии к правкам;
- историю кто, когда и что поменял;
- сравнение "было/стало";
- старую версию карточки перед изменением.

Клиентский отчет нужен. Удобные форматы: Excel, PDF и ссылка в сервисе.

## Интерфейс кабинета

Первый экран для специалиста:
- доступные кабинеты продавцов;
- названия кабинетов;
- задачи на сегодня;
- возможность добавить новый кабинет селлера;
- поиск по названию, а в будущем по маркетплейсу и названию.

Задачи:
- задачи на сегодня;
- приоритизация;
- высокий приоритет для цен и других рискованных изменений;
- просроченные задачи.

Таблица карточек:
- понятный веб-интерфейс;
- возможность скачать таблицу;
- фильтры по категории, статусу, ошибке, качеству, бренду, остаткам, цене и артикулу;
- подсветка карточек с большим количеством пустых или неправильно заполненных полей.

Массовые действия:
- редактирование любого столбца, кроме артикула;
- просмотр предложенных вариантов улучшения;
- возможность редактировать предложения в веб-версии без скачивания и повторной загрузки таблицы.

При клике на карточку должны открываться:
- расширенные данные;
- ссылка на карточку на маркетплейсе;
- процент/индикатор заполненности;
- проблемы, предложения и история изменений.

Статусы карточки:
- не проверена;
- есть проблемы;
- предложены правки;
- утверждена;
- обновляется;
- обновлена;
- ошибка.

Руководитель должен видеть:
- сводку состояния проектов;
- актуальные и просроченные задачи специалистов;
- количество задач по проекту;
- выполненное количество задач по проекту;
- команду проекта;
- признаки того, что проект забыт или застрял.

## Проверка по официальным материалам Wildberries

Дата проверки: 2026-07-07.

Официальная документация WB API подтверждает, что через API есть методы для работы с карточками товаров: создание, редактирование, получение карточек с ошибками, категории, подкатегории и характеристики. Также документация описывает токены, лимиты запросов и типовые HTTP-ошибки.

Полезные ссылки:
- WB API: работа с товарами - https://dev.wildberries.ru/docs/openapi/work-with-products
- WB API: общая информация, токены, ошибки и лимиты - https://dev.wildberries.ru/docs/openapi/api-information
- WB Partners: как создать карточку товара - https://seller.wildberries.ru/instructions/ru/by/material/how-to-create-a-product-profile-rb
- WB API release notes - https://dev.wildberries.ru/release-notes

Зафиксированные выводы:
- API-токен создается в настройках магазина и используется для доступа к WB API.
- Документация WB API указывает, что методы карточек покрывают создание, редактирование и получение ошибок.
- Для подкатегорий доступны методы получения списка подкатегорий и характеристик по `subjectId`.
- Наименование карточки ограничено 60 символами.
- Для актуальной интеграции нужно регулярно сверяться с release notes WB API, потому что поведение методов меняется.
- В release notes есть важные изменения по карточкам, фото и остаткам; например, в 2026 году менялась логика `withPhoto`, а для остатков акцент смещался с `sku` на `chrtId`.

## Продуктовые выводы после ответов

Главная ценность первой версии уточняется: не автопубликация любой ценой, а ускорение подготовки релевантных правок с обязательной ручной проверкой специалиста.

MVP должен сфокусироваться на:
- импорте карточек из Excel/CSV и позже через API;
- таблице карточек с фильтрами и подсветкой проблем;
- аудите заголовков, характеристик, описаний, цены и фото;
- генерации 2-3 вариантов заголовков/описаний/характеристик;
- объяснениях "плохо, потому что" и "предлагаю так, потому что";
- ручном утверждении правок специалистом;
- backup до изменений;
- сравнении "было/стало";
- журнале изменений;
- чек-апе после обновления;
- отчете клиенту.

Отдельно нужно спроектировать осторожную работу с ценами: цена важна и часто ошибается, но это зона максимального риска.

## Следующий экран прототипа

На основе ответов самый полезный следующий экран - детальная карточка товара с аудитом и предложениями.

Этот экран должен показать:
- исходные данные карточки;
- оценку заполненности;
- проблемы по заголовку, характеристикам, описанию, цене и фото;
- 2-3 варианта нового заголовка;
- предлагаемые характеристики;
- пояснения по каждому предложению;
- ручное редактирование;
- "было/стало";
- статус утверждения;
- кнопку применения только для технического специалиста;
- предупреждение для цены и обязательных атрибутов.

После него можно развивать массовый режим: выбранные карточки в таблице -> пакет предложений -> построчное утверждение -> публикация -> чек-ап.

## Что решено

- Первый маркетплейс - Wildberries.
- Сервис является внутренним рабочим пространством команды.
- Основной пользователь публикации - технический специалист.
- Менеджер не применяет изменения в карточках.
- Руководитель смотрит сводку проектов и задач.
- Все предложения требуют ручной проверки.
- В первую очередь автоматизируются заголовки, характеристики, описания и подбор семантики.
- Цена - зона повышенного риска.
- Нужны backup, история, "было/стало" и чек-ап после обновления.
- Интерфейс должен использовать корпоративные цвета WebOptimize.

## Добавление магазина

На главном экране при действии "Добавить магазин" пользователь должен выбрать способ подключения:

1. **Через API**.
   Пользователь вводит API-ключ Wildberries. Сервис проверяет доступ, сохраняет подключение и загружает карточки в режиме чтения.
2. **Ручной режим**.
   Пользователь заполняет базовые поля магазина: название магазина, маркетплейс, ответственная команда/менеджер, комментарий или другие минимальные данные. Карточки затем можно добавить через таблицы, список артикулов/nmID/SKU или другой ручной источник.

Внутри кабинета клиента/магазина должны быть доступны отдельные действия:
- включить или подключить API, если магазин изначально создан вручную;
- заменить или проверить API-ключ;
- добавить таблицу Excel/CSV с карточками;
- добавить выбранные карточки вручную по nmID, SKU, артикулу продавца или ссылке;
- видеть, из какого источника пришли данные: API, таблица или ручной ввод.

Продуктовое правило: API и ручной ввод являются разными входами в один и тот же кабинет магазина. После загрузки карточек анализ, категория, семантика, предложения, согласование и история должны работать одинаково независимо от источника.

Правило для первого API-этапа:
- только чтение данных;
- не создавать карточки;
- не редактировать карточки;
- не менять цены;
- не менять остатки;
- не отправлять массовые обновления;
- не выполнять публикацию;
- показывать, какую информацию API дает получить и как она выглядит в интерфейсе.

Backup нужно учитывать уже в архитектуре, но до разрешения на изменения backup остается проектным требованием, а не поводом выполнять запись в WB.

После переезда демо-данные интерфейса нужно удалить и заменить данными из реального кабинета WB, полученными через API в режиме чтения. Также планируется подключение API MPStats для анализа запросов, конкурентов, частотности и подготовки рекомендаций.

Уточнение по MPStats: MPStats API рассматривается не только как SEO-источник, но и как внешний источник витринных данных по магазину и карточкам - ссылки, видимые характеристики, конкуренты и позиции. Он может помочь собрать данные по публичной витрине магазина, но не заменяет WB API для владельческого доступа к кабинету, ошибок карточек и будущих операций записи.

### Реальные кейсы "было/стало"

Материалы:
- `document/кейс было стало описание.docx`;
- `document/кейс было стало заголовки.xlsx`;
- `document/кейс было стало заголовки 2.xlsx`.

Смысл кейсов:
- исправлялись заголовки и описания;
- добавлялись ключевые запросы;
- заголовки менялись на более высокочастотные ключи;
- для заголовков отдельно контролировалось количество символов;
- для описаний важно сравнение длинного текста "было" и нового текста "стало".

Вывод для интерфейса:
- в детальной карточке нужен блок "было/стало" для заголовка и описания;
- для заголовка нужен счетчик символов и предупреждение о лимите 60 символов;
- сервис должен показывать, какие ключи добавлены или усилены;
- желательно показывать причину изменения: более частотный запрос, релевантность товару, запрос конкурента, устранение нерелевантного слова.

### Инструкции, чек-листы и шаблоны

Текущих внутренних инструкций, чек-листов и шаблонов нет.

Вывод для сервиса: чек-лист качества карточки придется сформировать внутри продукта на основе опыта специалиста, материалов WB, MPStats и реальных кейсов.

### Ошибки и правила названий

Из базы знаний зафиксированы правила для названий:
- подобрать подходящие ключевые слова и добавить их в наименование;
- избегать синонимов;
- избегать дублирования;
- не использовать словосочетания, которые неверно описывают товар;
- лишние подробности, рекомендации к применению и оценочные суждения лучше переносить в описание.

Примеры ошибок из базы знаний:
- "Чехол для телефона смартфона прозрачный" - нужно убрать синоним;
- "Хайлайтер тени для век блестящая палетка для макияжа лица" - нужно убрать лишние подробности, рекомендации к применению и оценочные суждения.

Типовые ошибки/состояния карточек:
- ненормативная лексика;
- карточка не найдена, вероятно удалена;
- варианты характеристик с ошибками;
- новые характеристики без заполненных обязательных полей;
- нет фото в карточке;
- карточка в черновиках или в корзине.

Вывод для сервиса:
- нужен справочник типовых ошибок с понятным человеческим объяснением;
- ошибки WB должны отображаться рядом с полем, которое мешает обновлению;
- для каждой ошибки желательно показывать действие: исправить поле, добавить фото, заполнить обязательное, проверить статус карточки.

- Какой формат backup и отката нужен для первой версии.

- Какие категории товаров брать первыми для правил характеристик.

## Гипотеза MVP

Первая версия сервиса:

1. Внутренний веб-кабинет на поддомене.
2. Роли: менеджер, специалист, руководитель.
3. Первый маркетплейс: Wildberries.
4. Пользователь подключает магазин/API-ключ или передает конкретный список карточек.
5. Сервис загружает карточки по выбранному охвату: весь магазин или заданный набор.
6. Сервис проводит аудит по заданным правилам.
7. Сервис показывает оценку карточек и предложения по улучшениям.
8. Технический специалист вручную корректирует и утверждает предложения.
9. Перед изменениями сервис сохраняет старую версию карточки и показывает "было/стало".
10. Сервис применяет массовые изменения хотя бы для названий, характеристик и ключей.
11. Сервис проводит чек-ап после обновления и показывает статусы, ошибки и результат.
