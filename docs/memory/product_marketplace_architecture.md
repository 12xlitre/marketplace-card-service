# Архитектура маркетплейсов

Дата: 2026-07-16.

Устойчивое продуктовое решение:
- верхний уровень рабочего пространства - `Клиент`;
- внутри клиента есть разделы маркетплейсов, сначала `Wildberries`, затем `Ozon`;
- у каждого маркетплейса свой набор кабинетов, карточек, задач и отчетных периодов;
- текущий WB-поток остается базовым рабочим сценарием и не должен ломаться при добавлении Ozon;
- Ozon стартует как скрытая beta-вкладка только для Дмитрия.

Текущая реализация первого шага:
- отдельная backend-сущность `Client` пока не введена: у `portals` уже есть поле `marketplace`, а `POST /api/portals` по-прежнему поддерживает рабочий API-сценарий только для Wildberries;
- экран `Кабинеты` во frontend показывает список клиентов, а не общий ряд маркетплейсов;
- вход в клиента открывает общий экран клиента с редактируемым именем, командой проекта, контактными данными и кнопками разделов `Wildberries` / `Ozon`;
- если у старого WB-кабинета нет явного `clientName`, frontend не должен показывать технический fallback `Кабинет WB` как имя клиента; такой клиент показывается как `Клиент без названия` и переименовывается из шапки клиента;
- до появления backend-сущности `Client` редактирование имени клиента сохраняет `clientName` во все кабинеты этого клиента через `POST /api/portals/<portal_id>/client-name`;
- до появления backend-сущности `Client` редактирование команды клиента сохраняет выбранных `Руководитель отдела` / `Технический специалист` / `Аккаунт-менеджер` во все кабинеты этого клиента;
- до появления backend-сущности `Client` редактирование контактов клиента сохраняет `Контактное лицо` / `Телефон` / `Email` / `Комментарий` во все кабинеты этого клиента через `portals.client_contact_json`;
- архивирование и удаление клиента пока остаются bulk-операциями над кабинетами клиента: удалить клиента можно только после архива всех его кабинетов;
- раздел `Wildberries` внутри клиента использует существующие карточки кабинетов и открывает прежний `SellerScreen`;
- если у клиента ровно один кабинет маркетплейса, кнопка маркетплейса открывает его сразу; список выбора кабинета показывается только при 0 или 2+ кабинетах;
- раздел `Ozon beta` виден только пользователю Дмитрию по логину/ФИО и создает тестовый manual Ozon-кабинет внутри текущего клиента через форму подключения с обязательным источником: ссылка на кабинет/продавца, Seller ID, SKU или список товаров; отдельное поле тестовых артикулов/SKU ограничивает первую MPStats-проверку, чтобы не тянуть всю витрину сразу;
- Ozon-кабинет, как и WB, создается с охватом `full` или `selected`; по умолчанию в beta выбран `selected`, а поле `Артикулы или SKU` обязательно только для выбранных карточек;
- открытие Ozon-кабинета ведет в отдельный Ozon beta-экран, визуально похожий на WB-кабинет, но без WB API и без подмешивания WB-задач/отчетов;
- в Ozon beta добавлен сохраняемый ручной источник кабинета: ссылка/Seller ID/ориентир Ozon и комментарий хранятся в `portals.store_url` / `portals.manual_source` через `POST /api/portals/<portal_id>/manual-source`, а статус маршрута Ozon меняется с `ожидает источник` на `источник задан`;
- Ozon beta умеет вручную проверить сохраненный источник через MPStats (`POST /api/portals/<portal_id>/ozon-mpstats-probe`): это read-only probe с агрегацией найденных SKU до 50 карточек, sample-карточками и списком попыток endpoint-ов, без сохранения карточек и без запуска WB bootstrap;
- найденные Ozon sample-карточки сохраняются только отдельным явным действием через `POST /api/portals/<portal_id>/ozon-mpstats-cards`; они попадают в `cards_snapshot_json` как `marketplace=ozon` и отображаются в Ozon-разделе, но пока не открываются в WB-деталке;
- список сохраненных Ozon-карточек повторяет рабочий контур WB-кабинета: статус-бар, поиск, фильтры, таблица с SKU/offer, заполненностью, замечаниями, статусом работы и переходом в детали, но все признаки считаются только по Ozon snapshot;
- сохраненные Ozon-карточки открываются в отдельный `OzonCardDetailScreen`, который повторяет структуру WB-detail, но пока работает в режиме просмотра Ozon snapshot: вкладки СЯ/аудит/изменения являются Ozon-заглушками и не запускают WB-логику;
- модалка добавления кабинета пока оставлена WB-only, чтобы не запускать Ozon через WB/MPStats-логику.

Правило на следующие шаги: до подключения Ozon API нужно явно разделить marketplace-specific контракты загрузки карточек, задач, отчетных периодов и внешних источников. Нельзя смешивать Ozon-данные в WB-потоке только через общий `portal_id` без проверки `marketplace`.

Уточнение по Ozon и MPStats:
- для Ozon планируется похожая связка с MPStats, как в WB-потоке;
- карточки и кабинетный источник должны быть Ozon-specific, но семантическое ядро для Ozon собирается через MPStats по ключам WB;
- поэтому модуль СЯ должен различать `marketplace` карточки и `keywordSourceMarketplace`: для Ozon это может быть `marketplace=Ozon`, `keywordSourceMarketplace=Wildberries`;
- нельзя жестко переименовывать все MPStats/СЯ сущности в Ozon-specific: часть аналитики останется WB keyword-базой, даже когда рабочий кабинет Ozon.
