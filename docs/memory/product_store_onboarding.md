# Заведение портала/магазина

Дата: 2026-07-07.

Устойчивое продуктовое решение:
- на главном экране действие "Добавить портал" должно давать выбор: "Через API" или "Ручной режим";
- через API пользователь вводит ключ Wildberries, сервис проверяет доступ и загружает карточки в режиме чтения;
- в ручном режиме пользователь создает портал по базовым полям, например название, маркетплейс, ответственная команда/менеджер и комментарий;
- внутри кабинета портала API можно подключить позже;
- внутри кабинета портала можно добавлять Excel/CSV-таблицы и выбранные карточки вручную по nmID, SKU, артикулу продавца или ссылке.

Архитектурное правило: API, таблицы и ручной ввод являются разными входами в одну модель портала/магазина и карточки. После загрузки карточек анализ, категория, семантика, MPStats, предложения, согласование и история должны работать одинаково независимо от источника.

Ограничение текущего статического прототипа: форма API нужна для проверки UX, но ключ Wildberries не сохраняется в браузере и не отправляется напрямую из frontend. Реальное подключение должно идти через backend с безопасным хранением токена и read-only проверкой доступа.

Уточнение по источникам:
- WB API - основной источник владельческого доступа к кабинету: список карточек продавца, nmID, vendorCode, характеристики, ошибки и будущие операции записи после отдельного разрешения.
- MPStats API может быть отдельным источником витринных данных по магазину и карточкам: ссылки/позиции карточек, видимые характеристики, конкуренты, частотность и SEO. Он полезен для аудита и обогащения, но не заменяет WB API для управления кабинетом продавца.
- link-based сценарий нужен как быстрый вход без WB API: пользователь вставляет ссылку на магазин WB или список ссылок на карточки, backend извлекает `sellerId`/`nmID`, подтягивает публичные данные через MPStats и создает такой же кабинет/карточки, как при API-загрузке.

Решение на 2026-07-08:
- первые backend-шаги: серверный запуск приложения и encrypted storage WB-токена;
- WB API ключ не вводить в браузере и не передавать в чат; добавлять через `.env`/серверную команду;
- при подключении WB API загружаем весь магазин, а затем внутри кабинета выбираем рабочий набор карточек по nmID/SKU/фильтрам;
- название магазина должно определяться после API-проверки по данным WB, временное имя допускается только до первой загрузки;
- первый этап интеграции строго read-only: без обновления карточек, цен, остатков и публикации.

Приоритет на 2026-07-09:
- добавить второй способ заведения магазина: по ссылке на магазин WB и/или по ссылкам на карточки;
- понять, какие текущие поля карточки можно надежно тянуть из MPStats `/items/{nmID}` и `/items/{nmID}/full`, а какие остаются только аналитическими сигналами;
- обеспечить, чтобы audit, `Взять в работу`, черновики и согласование работали одинаково для WB API и link-based кабинетов.

Уточнение на 2026-07-13:
- для ссылки вида `https://www.wildberries.ru/seller/{sellerId}` первичным источником карточек должен быть публичный WB seller catalog (`wb-public-seller`), а сохраненный снимок считается подходящим только если карточки имеют тот же `supplierId`;
- публичные фото WB завязаны на диапазоны `basket-*.wbbasket.ru`, которые меняются со временем; при link-based загрузке нужно хранить `photos/photoUrl`, но UI должен уметь восстановить URL по `nmID`, если старый сохраненный `basket` оказался битым.
- при дозагрузке seller-витрины нельзя полагаться только на `existing_count // 100 + 1`: WB может отдавать нестабильные страницы/429, поэтому безопаснее сканировать витрину с первой страницы, пропускать уже сохраненные `nmID` и добирать новые;
- функцию загрузки нельзя заставлять мутировать исходный set сохраненных ключей (`existing_keys`): если найденные новые `nmID` добавить туда слишком рано, следующий этап дедупликации ошибочно выкинет их как уже существующие;
- frontend должен предпочитать сохраненные backend `photos/photoUrl` перед расчетным `basket` fallback, потому что расчет по диапазонам может быть на соседнюю корзину мимо, а backend уже проверяет реальные URL.
