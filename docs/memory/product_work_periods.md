# Work Periods

Дата: 2026-07-14.

Устойчивое продуктовое решение:
- `Отчетный период` - это вкладка внутри конкретного кабинета селлера с планом работ отдела, а не клиентская XLSX-выгрузка из вкладки `Отчеты` и не MPStats-период;
- период создается карточкой/квадратом по сценарию, похожему на создание кабинета: пользователь нажимает `Создать`, задает название и произвольные даты начала/конца через календарь;
- даты не привязаны к календарному месяцу, потому что работа может начаться в любой день;
- при создании пользователь выбирает чек-лист конкретных работ Wildberries из полного списка планирования, сгруппированного по смыслу: аналитика и стратегия, карточки и контент, отзывы и лояльность, реклама и трафик, витрина и продвижение магазина, поставки/остатки/экономика, отчеты;
- базовый список создан 2026-07-15 по файлу пользователя `document/План работ ВБ все задачи.docx`: 36 детальных пунктов без дословных дублей (`A/B тест инфографики` и `Предложение по подготовке Рич-контента` встречались повторно);
- старые укрупненные ключи `semantic/content/prices/stocks` остаются допустимыми только для совместимости уже созданных периодов, но новые периоды по умолчанию создаются из детального списка;
- пока период идет, план можно корректировать: добавлять активные пункты из списка и убирать ненужные;
- убранный пункт не удаляется бесследно, а получает статус `excluded`, причину/дату исключения и остается в истории/выгрузках;
- каждый пункт плана хранит рабочий статус `planned/in_progress/review/done`, служебные статусы `returned/excluded`, комментарий, дату/исполнителя последней смены статуса, дату выполнения, исполнителя, причину возврата, дату возврата, причину исключения и историю действий;
- экран созданного периода показывает выбранные работы не плоским списком, а группами из чек-листа; у группы есть счетчик выполнения/возвратов/исключений, а действия по пункту раскрываются только по кнопке `Работать`;
- у пункта плана используются необязательные поля связи с задачами кабинета (`linkedTaskIds`, `linkedBatchIds`): пачку можно привязать при создании через `Взять в работу` или позднее кнопкой `Привязать к плану`; план должен работать и без такой связи;
- статус фиксируется явным действием менеджера/специалиста: `В плане`, `В работе`, `На согласовании`, `Выполнено`; при сохранении статуса сохраняются комментарий, дата, исполнитель и приложенный файл до 2 МБ;
- возврат обязателен с причиной, старые данные выполнения не стираются без истории;
- сразу после создания/корректировки плана можно скачать презентабельный XLSX `План работ` для клиента;
- финальный XLSX-отчет открывается только после даты окончания периода; до этого API `generate_report` должен fail-closed с `work_period_not_finished`;
- в конце периода пользователь скачивает итоговый отчет: исходный план, статусы, даты закрытия задач, что не выполнено или возвращено, и почему; XLSX должен иметь управленческую сводку, лист по разделам и отдельные листы `Не выполнено`, `Выполнено`, `Исключено`;
- после формирования итогового отчета карточка периода показывает экранную сводку: кто/когда сформировал, выполнено/не выполнено/возвраты/исключено и краткие причины по группам;
- данные хранятся в backend-таблице `portal_work_periods`, строго в рамках `portal_id`;
- маршрут `GET/POST/DELETE /api/portal-work-periods` обязан проверять доступ пользователя к кабинету через `user_can_access_portal`.
- backend action `link_task` внутри `POST /api/portal-work-periods` связывает реальную задачу/пачку с пунктом плана, двигает эту связь внутри выбранного периода при повторной привязке и переводит пункт из `planned` в `in_progress`.
