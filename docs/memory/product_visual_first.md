# Visual-first подход

Дата: 2026-07-06.

Пользователь явно сказал, что работает глазами: ему важно видеть интерфейс, кнопки, статусы и экраны, а не только логическую схему.

Договоренность:
- сначала делать понятный визуальный прототип рабочего процесса;
- затем уточнять логику и код;
- идти по шагам от фактических ответов на discovery-вопросы;
- постоянная локальная точка входа прототипа: `http://localhost:5173/` после запуска `npm run prototype`.

Текущая продуктовая рамка:
- кабинет сотрудников;
- переход в кабинет селлера;
- поддержка двух режимов охвата: полный магазин и выбранные карточки.

Обновление от 2026-07-07:
- после ответов технического специалиста следующий экран прототипа выбран и реализован как детальная карточка товара в `index.html`;
- ключевая логика экрана: аудит карточки, 2-3 варианта правок, объяснение "плохо/предлагаю", ручное утверждение специалистом, "было/стало", backup и предупреждения для цены/обязательных атрибутов;
- менеджер не применяет изменения, технический специалист является основным пользователем публикации.
- доступны реальные материалы в `document/`: пример выгрузки WB, кейсы "было/стало" по заголовкам/описанию и ссылки на хорошую/плохую карточку в `docs/discovery.md`.
- вход в продукт должен начинаться с логина/пароля; пользователи хранятся в серверной базе, публичный CSV с паролями удален.
- тестовый WB API до отдельного решения используется только для чтения и отображения данных; любые изменения в Wildberries запрещены.
- при переходе от прототипа к сервису тестовые пользователи, CSV и демо-данные не переносим; делаем реальные доступы, реальный WB read-only и затем подключаем MPStats API.
