# Semantic Core

Дата: 2026-07-14.

Устойчивое решение:
- вход во вкладку `Семантическое ядро` сначала показывает стабильные ранжирующиеся запросы карточки из MPStats `/analytics/v1/wb/items/{nmID}/keywords`; смена стартового SEO-запроса не должна менять этот набор, а должна менять только кандидатов к добавлению;
- `Действующие ключи` нельзя автоматически формировать из отдельных слов и фрагментов характеристик: характеристика может описывать состав, назначение или упаковку, но это не доказывает поисковый спрос. Ключ попадает в СЯ только если есть строка MPStats с частотностью/позициями и запрос проходит товарную релевантность; фрагменты вроде `нарушения углеводного обмена` и `одной упаковке блистера леденцов` отбрасываются.
- `Ключи в карточке` не должны определяться мягким совпадением отдельных слов по всему описанию: query считается уже заложенным в контент только при строгом фразовом совпадении или короткой двухсловной перестановке;
- SEO expansion MPStats должен сохранять полный рабочий набор `allKeywords` до 5000 строк, иначе высокочастотные, но не первые в текущем срезе запросы исчезают из правой колонки;
- метрики MPStats в СЯ нельзя склеивать через fallback: прямой `wb_count` является `частота WB`, а `wbcount` из SEO expansion, `wb_cluster_count` и `norm_query_count` не должны подписываться как частота запроса. Если прямой WB-счетчик равен 0 или пустой, кластерное/расширенное число не должно превращаться в частоту WB;
- стартовый запрос MPStats можно расширять дополнительными seed-фразами из самой карточки, если они явно читаются в названии, описании или характеристиках; пример: для описания `очки кошки` добавляем `очки кошачий глаз`, чтобы получить cat-eye запросы;
- оператор может заменить стартовый запрос в поле СЯ и нажать Enter: это запускает новый подбор по введенной фразе и меняет только кандидатов к добавлению; для карточек очков ручной seed без слова `очки`, например `кошачий глаз`, дополнительно квалифицируется товарными seed-фразами `очки ...` / `солнцезащитные очки ...`;
- архивные подборки СЯ являются отдельной кабинетной библиотекой: применение копирует ключи в карточку, пополнение добавляет в архив только новые ключи из карточки, а удаление из архива не меняет уже сохраненные карточки;
- во вкладке `Изменения -> Контент` подсветка ключей в черновике описания должна быть отдельным превью рядом с textarea: специалист видит связь фрагмента текста с ключом, но редактирование остается в обычном поле без потери курсора.
- дополнительные seed-фразы требуют простого товарного guard: для карточек про очки кандидаты должны содержать корень `очк`, чтобы не протаскивать нерелевантные строки вроде `черная кошка`;
- поле `Слова исключения` в СЯ принимает до 10 уникальных слов через пробел, запятую или точку с запятой; лишние слова не участвуют в фильтрации кандидатов;
- частотные корзины по умолчанию: высокий `>=5000`, средний `>=2000` и `<5000`, низкий `<2000`; backend-пороги задаются `MPSTATS_SEMANTIC_HIGH_FREQUENCY` и `MPSTATS_SEMANTIC_MEDIUM_FREQUENCY`, автодобавление берет максимум 36 запросов, до 12 из каждой корзины.
- действующие `Ключи в карточке` можно пометить `к удалению`: это предложение для согласования, а не автоматическая правка WB. Итоговый отчет СЯ показывает такие ключи отдельными колонками `Ключ к удалению из карточки`, `Причина удаления`, `Согласование удаления`; после согласования специалист вручную удаляет ключ в SEO карточки WB и проводит переоптимизацию без него.
- кнопка `Переоптимизировать` во вкладке СЯ должна отдавать LLM все целевые ключи: добавленные, действующие в контенте и ранжируемые, исключая `к удалению`; результат должен переписывать `title`, почти до лимита 2000 символов расширять `description` и возвращать черновик подходящих характеристик.
- во вкладке `Изменения -> Контент` переоптимизация должна быть доступна отдельно у каждого блока: `Еще вариант заголовка`, `Еще вариант описания`, `Еще вариант характеристик`; частичный запуск не должен менять и даже переотмечать source/reason у остальных блоков.
- заголовок после переоптимизации формируется по WB-формуле `Тип товара -> Назначение -> Целевая аудитория -> Главная характеристика -> Дополнительная характеристика -> Бренд -> Модель`: начинать с типа товара, не с бренда/артикула; цвет и материал включать только если они реально помогают выбору, для оправ обычно сильнее `женская` + форма + бренд.
- переоптимизация обязана заново определять тип товара по `subjectName`/title/description каждой карточки: `солнцезащитные очки` не превращаются в `оправу`, даже если в характеристиках есть `Материал оправы` или в расширенных кандидатах встречаются ключи про оправу; для таких карточек title начинается с `Солнцезащитные очки`, а описание пишет про защиту от солнца/бликов/линзы/посадку без сценария установки диоптрийных линз.
- backend переоптимизации не должен полагаться только на payload открытой вкладки: перед LLM он добирает сохраненное СЯ из `card_drafts.meta.semanticCoreFinal/semanticCoreReports`, чтобы карточки вроде `397277226` не сжимались до одних `rankedKeywords`.
- при объединении target-ключей с расширенными title-кандидатами backend сохраняет `semanticSource`: пол/аудитория для заголовка выбираются из фактов и целевых `added/current` ключей, а не из высокочастотного хвоста рекомендаций без согласования.
- описание после переоптимизации не должно превращаться в список ключей: сначала анализ товара и покупателя, затем абзацы "что это", "позиционирование модели", "назначение и линзы", "дизайн/цвет/форма", "материалы/комплект", "бренд/итог"; ключи внедряются по одному разу в смысловые фразы, а не механическими повторами.
- если точный ключ звучит неестественно, нормальный русский язык важнее счетчика: запрос можно адаптировать или пропустить с предупреждением; запрещены конструкции вроде `вариант формата оправа...`, `Доступные оттенки` для цвета одной модели, утверждения о коррекции зрения самой оправой и неподтвержденные общие обещания (`долговечность`, `качественные материалы`, `не будет перегружать образ` без факта дизайна).
- черновик характеристик после переоптимизации должен схлопывать близкие названия в каноническое WB-поле, если совпадение сильное или `charcID` одинаковый; пример: `Конструкции оправы` не добавляется отдельным свободным полем, а мержится в существующую `Конструкция оправы`.
