# WB Characteristic Values

Дата: 2026-07-08.

Устойчивое решение:
- официальный WB API `GET /content/v2/object/charcs/{subjectId}` используется для списка характеристик категории и метаданных `maxCount`, `charcType`, `required`, `hasFilter`, `isVariable`, `existNamedField`;
- этот метод не дает полный список допустимых значений для произвольной характеристики;
- полные directory-справочники через WB API подключаются только для отдельных полей: цвет, пол/гендер, страна производства, сезон, НДС, ТНВЭД;
- для остальных характеристик UI не должен называть подсказки "полным списком WB".

Правило интерфейса:
- строгие WB-справочники показываются как `значения из справочника WB`, свое значение в них не добавляется;
- свободные характеристики показываются как `подсказки из карточек` и `свое значение разрешено`;
- если список построен из карточек кабинета, нужно явно писать, что это не полный список WB.

Следующий источник данных:
- MPStats подключается как внешний источник подсказок по `subjectID + charcID`, вместе с накопительным словарем ручных значений специалистов;
- соединение MPStats проверяется backend-запросом к официальному WB analytics API с заголовком `X-Mpstats-TOKEN`; токен хранится зашифрованно и не возвращается во frontend.
- первый MPStats-слой использует отчет `/analytics/v1/wb/characteristics-analysis`: результат нормализуется по названиям характеристик и подмешивается в текущие чипсы-подсказки; следующий риск — несовпадение названий MPStats и WB `charcID`, для этого может понадобиться маппинг.
- карточный аудит вынесен в backend route `POST /api/card-audit`: результат строится по методике OptiCards/MP Audit Сергея, через `evidenceBundle`, JSON-блоки `category/competitors/title/description/characteristics/summary` и `riskNotes` при нехватке данных.
- LLM допускается только как refinement поверх собранных фактов; без LLM-ключа deterministic audit все равно возвращает draft-предложения и не должен выдумывать отсутствующие данные.
- сопоставление характеристик MPStats к WB должно быть строгим: слабого token overlap недостаточно, потому что он давал ложные пары вроде `Тип ростовки` -> `Тип карманов` и `Размер на модели` -> `Материал`;
- технические ошибки MPStats не должны попадать в UI как есть: если рыночный контекст ниши не загрузился из-за отсутствующего `path`, пользователь видит понятное ограничение аудита, а не endpoint и текст API-валидации.
- публичные фильтры WB-каталога рассматривались как обходной вариант, но не выбраны как основа: источник может быть нестабильным, не является seller API и потребует маппинга фильтров каталога к `charcID`.
