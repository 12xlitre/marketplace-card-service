# Mass Workflow

Дата: 2026-07-08.

Устойчивое продуктовое решение:
- массовая работа не должна превращаться в локальную правку одной карточки через WB API;
- базовый сценарий: выбрать карточки в кабинете селлера -> `Взять в работу` -> технический специалист проходит карточки по одной -> аккаунт-менеджер проверяет пакетной сводкой;
- режим технического специалиста должен быть конвейером: одна карточка на экране, предложения по улучшению, быстрые действия `принять`, `править`, `пропустить`, `готово, следующая`, прогресс по пакету;
- при открытии активной пачки из вкладки `Задачи` детальная карточка показывает контекст пачки: тип работы, прогресс `N из M`, текущую карточку и кнопки `Предыдущая`/`Следующая`, чтобы специалист проходил пакет без возврата в список;
- порядок карточек внутри пачки можно менять вручную во вкладке `Задачи`: frontend дает drag-and-drop редактор, при удержании у верхнего/нижнего края экрана сам прокручивает длинный список, держит перестановки локально до явного `Сохранить порядок`, backend хранит порядок в `meta.batch.position`, а переходы `Предыдущая`/`Следующая` идут по сохраненному порядку;
- конвейер поддерживает быстрые действия `готово, следующая`, `пропустить`, `вернуться позже`, фильтры строк пачки и журнал событий; пакетный аудит запускается из пачки последовательно по видимым карточкам, сохраняет audit/content draft, но не отправляет секции на согласование автоматически;
- режим аккаунта должен быть сжатым проверочным экраном по пакету: что изменилось, риски, статус, возможность открыть деталь, принять пакет или вернуть на доработку;
- после принятия первым релизным действием остается выгрузка таблиц WB, а не write-запрос в Wildberries;
- кнопка прямой отгрузки в WB допустима только как будущий этап и должна быть отключена до отдельного решения по write-операциям.
- пачка работ может включать отдельные типы `СЯ`, `Контент`, `Цены`, `Остатки`; задача по СЯ не обязана проходить approvalSections контента и считается закрытой, когда специалист явно сохраняет карточку в итоговое СЯ кабинета.
- задача по `Контенту`, `Ценам` или `Остаткам` считается закрытой для активной пачки специалиста после отправки соответствующей секции на согласование (`submitted`), принятия (`approved/exported`) и возвращается в пачку только при `changes_requested`;
- связанный пункт отчетного периода агрегируется по всей пачке: частично закрытая пачка остается `В работе`, полностью отправленная на согласование по `Контенту`/`Ценам`/`Остаткам` становится `На согласовании`, полностью принятая/выгруженная или полностью финализированная по СЯ становится `Выполнено`;
- задачей кабинета считается только `card_drafts.meta.batch.workTypes`; обычное сохранение СЯ, истории MPStats или черновика карточки без batch не должно создавать задачу `Контент`;
- при `Взять в работу` можно выбрать пункт активного отчетного периода; backend привязывает созданный `batchId` и тип задачи к `portal_work_periods.tasks_json`, а уже существующую пачку можно привязать или отвязать позднее из вкладки `Задачи`;
- Ozon beta следует тому же UX пачек, но с отдельным источником данных: выбранные Ozon-карточки создают массовую задачу с `batchId` и `workTypes=["semantic","content"]`, вкладка `Задачи` группирует ее по секциям, показывает прогресс СЯ/контента, статусы `на согласовании/согласовано/возврат/позже` и дает привязать пачку к нескольким пунктам отчетного периода через общий `portal_work_periods`;
- ошибочную или тестовую задачу можно удалить через `POST /api/card-workset/delete-tasks`: удаляется рабочий batch/тип работ, matching-связь с отчетным периодом очищается, но сохраненные результаты карточки остаются в `card_drafts`.
- при открытии карточки из пачки нельзя матчить задачу по пустым полям (`vendorCode=""` и т.п.): сначала `cardKey`/`nmID`, затем непустой артикул. Иначе задача может открыть первую карточку кабинета и сохранить результат вне batch.
- закрытая работа не исчезает бесследно: `GET /api/approval-workflow` отдает `completedTasks`, а вкладка `Задачи` показывает блок `Завершенные` с поиском, типом работы, датой/временем, исполнителем и возможностью снова открыть карточку.

Следующий рекомендуемый шаг:
- сделать пакетный запуск аудита для выбранных карточек после `Взять в работу`;
- добавить быстрые действия в конвейер специалиста: `готово, следующая`, `пропустить`, фильтр только незавершенных карточек пачки.
