# Approval Workflow

Дата фиксации: 2026-07-08.

Роли процесса:
- технический специалист готовит черновик изменений;
- аккаунт-менеджер проекта согласовывает: `Принято` или `На доработку` с обязательной причиной;
- руководитель отдела наблюдает процесс и статистику, но не редактирует карточку и не согласует вместо аккаунт-менеджера;
- админ может технически выполнить согласование.

Хранение:
- общий статус задачи живет в `card_drafts.meta.approval`, статусы блоков `Контент`/`Цены`/`Остатки` живут в `card_drafts.meta.approvalSections`;
- переходы статусов пишутся в `card_approval_events` для аналитики и не удаляются при сбросе черновика;
- `GET /api/approval-workflow?portal_id=...` отдает текущие задачи, историю и простую аналитику по кабинету.

Совместимость и права:
- старые/частичные черновики без `approvalSections` сравниваются с fallback из общего `meta.approval`, чтобы отправка на согласование не блокировалась соседними секциями;
- `submitted` может ставить доступный участник кабинета, `approved` и `changes_requested` может ставить только аккаунт-менеджер проекта или админ;
- `POST /api/card-drafts` возвращает различимые ошибки `approval_forbidden`, `invalid_json`, `draft_payload_too_large`, чтобы UI показывал конкретную причину отказа backend.

Обновление WB:
- обычная кнопка `Загрузить свежие данные` не удаляет черновик и задачу согласования;
- обновление WB выполняет soft reset анализа через `POST /api/portals/{id}/reset-analysis-cache`;
- soft reset очищает MPStats-кэш и помечает audit-черновики как `stale`, но сохраняет значения в колонке `Стало` и `meta.approval`.

UX задач:
- С 2026-07-21 в разделе `Контент` пачки показывают одну основную контентную кнопку вместо двух похожих действий: при наличии итогового СЯ она называется `Доделать контент`/`Подготовить контент по СЯ` и запускает `/api/card-workset/reoptimize-content-task` по всей подходящей пачке, обновляя заголовок, описание и характеристики; если итогового СЯ еще нет, кнопка использует первичную подготовку черновиков.
