# Marketplace Card Service

## Идея

Внутренний сервис для компании, которая ведет клиентов-продавцов на маркетплейсах.

Проблема: нельзя сразу брать в работу большое количество карточек магазина, но малый объем карточек часто не устраивает клиентов. Пример запроса: 1000 карточек.

Предполагаемая задача сервиса:
- заходить в магазин/кабинет клиента;
- получать информацию по всем товарным позициям;
- массово готовить и вносить изменения по согласованию;
- работать с заголовками, описаниями, ключами и другими полями карточек.

## Формат продукта

Сервис должен быть визуальным рабочим пространством для технического специалиста, а не набором разрозненных скриптов.

Предполагаемый формат:
- отдельный кабинет/веб-приложение, возможно на поддомене;
- внутри кабинета специалист видит клиентов, магазины, карточки, задачи, статусы и результаты;
- действия по массовой обработке карточек должны быть управляемыми через интерфейс;
- сервис должен помогать контролировать путь от загрузки данных до согласования и публикации изменений.

## Текущий прототип

Frontend переведен на React + Vite. Исходники лежат в `frontend/`, production artifact собирается single-file в корневой `index.html`, чтобы текущий prod Dockerfile продолжал работать без изменения серверной обвязки.

```bash

```

Команды frontend:

```bash

npm run frontend:build

```

В прототипе есть:
- экран логина через backend: пользователи в SQLite, пароли в PBKDF2-хешах, сессия в HttpOnly cookie;
- read-only backend-подключение WB API для загрузки списка карточек через `/api/wb/cards`;
- главный экран `Кабинеты` с демо WB кабинетом, реальными кабинетами из backend, фильтром активных/неактивных и добавлением портала;
- левое меню `Кабинеты`, `Аудит`, `Настройки`;
- переход в кабинет селлера/портала;
- обзор кабинета: карточки, карточки к проверке, честный маршрут работы, участники проекта;
- состав проекта из серверного списка пользователей;
- добавление портала через API или вручную; WB API ключ отправляется только в backend, проверяется read-only запросом и сохраняется зашифрованно;
- режимы охвата работы: полный магазин и выбранные карточки;
- визуальный маршрут загрузка -> аудит -> правки -> согласование -> обновление;
- таблица карточек, статусы, ошибки WB и быстрые действия.
- детальная карточка с вкладками `Карточка`, `Аудит`, `Изменения`;
- вкладка `Аудит` принимает необязательные ссылки/nmID конкурентов именно для текущего аудита, не смешивая их с отдельным блоком `ТОП конкурентов`;
- вкладка `Отчеты` внутри кабинета селлера с первым клиентским WB XLSX-отчетом и явным выбором периода перед выгрузкой;
- ручной черновик "Было / Стало" без обязательного аудита: заголовок, описание, характеристики;
- редактирование характеристик через чипсы значений в колонке `Стало`, с лимитами WB `maxCount/charcType`, подсказками и возможностью удалить/добавить значения;
- пометки источников подсказок характеристик: строгий справочник WB, подсказки из карточек, свободное значение.

## Следующий шаг

Ответы пользователя на вопросы из `docs/discovery.md` разобраны и внесены в discovery.

Текущий продуктовый шаг: довести качество backend-аудита с MPStats до стабильного рабочего инструмента и открыть быстрый сценарий добавления магазина без WB API — по ссылке на магазин или карточки.

После read-only подключения WB: развить массовый режим вокруг детальной карточки - взять выбранные карточки в работу, пакетно запустить аудит, подготовить черновики `Было / Стало`, согласовать и выгрузить результат.

Важное решение: публичные тестовые учетные записи убраны из frontend. После стабилизации auth продолжаем подключать реальный WB API только на чтение и затем MPStats API для анализа.

Основной discovery-документ: `docs/discovery.md`.
