# Marketplace Card Service

## Идея

Внутренний сервис для компании, которая ведет клиентов-продавцов на маркетплейсах.

Проблема: нельзя сразу брать в работу большое количество карточек магазина, но малый объем карточек часто не устраивает клиентов. Пример запроса: 1000 карточек.

Предполагаемая задача сервиса:
- заходить в магазин/кабинет клиента;
- получать информацию по всем товарным позициям;
- массово готовить и вносить изменения по согласованию;
- работать с заголовками, описаниями, ключами и другими полями карточек.

## Формат продукта

Сервис должен быть визуальным рабочим пространством для технического специалиста, а не набором разрозненных скриптов.

Предполагаемый формат:
- отдельный кабинет/веб-приложение, возможно на поддомене;
- внутри кабинета специалист видит клиентов, магазины, карточки, задачи, статусы и результаты;
- действия по массовой обработке карточек должны быть управляемыми через интерфейс;
- сервис должен помогать контролировать путь от загрузки данных до согласования и публикации изменений.

## Текущий прототип

Frontend переведен на React + Vite. Исходники лежат в `frontend/`, production artifact собирается single-file в корневой `index.html`, чтобы текущий prod Dockerfile продолжал работать без изменения серверной обвязки.

Прод-адрес:

```text
https://opticards.weboptai.ru/
```

Health:

```text
https://opticards.weboptai.ru/healthz
```

Локальная ссылка после запуска:

```text
http://localhost:5173/
```

Команда запуска:

```bash
npm run prototype
```

Команды frontend:

```bash
npm run frontend:dev
npm run frontend:build
npm run frontend:check
```

В прототипе есть:
- экран логина через backend: пользователи в SQLite, пароли в PBKDF2-хешах, сессия в HttpOnly cookie;
- read-only backend-подключение WB API для загрузки списка карточек через `/api/wb/cards`;
- главный экран `Кабинеты` с демо WB кабинетом, реальными кабинетами из backend, фильтром активных/неактивных и добавлением портала;
- левое меню `Кабинеты`, `Аудит`, `Настройки`;
- переход в кабинет селлера/портала;
- обзор кабинета: карточки, карточки к проверке, честный маршрут работы, участники проекта;
- состав проекта из серверного списка пользователей;
- добавление портала через API или вручную; WB API ключ отправляется только в backend, проверяется read-only запросом и сохраняется зашифрованно;
- режимы охвата работы: полный магазин и выбранные карточки;
- визуальный маршрут загрузка -> аудит -> правки -> согласование -> обновление;
- таблица карточек, статусы, ошибки WB и быстрые действия.
- детальная карточка с вкладками `Аудит`, `Характеристики`, `Изменения`;
- ручной черновик "Было / Стало" без обязательного аудита: заголовок, описание, характеристики;
- редактирование характеристик через чипсы значений в колонке `Стало`, с лимитами WB `maxCount/charcType`, подсказками и возможностью удалить/добавить значения;
- пометки источников подсказок характеристик: строгий справочник WB, подсказки из карточек, свободное значение.

## Следующий шаг

Ответы пользователя на вопросы из `docs/discovery.md` разобраны и внесены в discovery.

Следующий продуктовый шаг: использовать MPStats как источник расширенных подсказок значений характеристик и данных для аудита карточек.

После read-only подключения WB: развить массовый режим вокруг детальной карточки - пакет предложений, построчное утверждение, публикация и чек-ап результата.

Важное решение: публичные тестовые учетные записи убраны из frontend. После стабилизации auth продолжаем подключать реальный WB API только на чтение и затем MPStats API для анализа.

Основной discovery-документ: `docs/discovery.md`.

- для характеристик учитываются `maxCount` и `charcType`: для `charcType = 4` лимит считается как 1 значение, для `maxCount > 0` показывается `X из N`;
- backend route `GET /api/wb/characteristics?portal_id=...&subject_id=...` загружает характеристики категории WB и возвращает `charcID`, `name`, `required`, `maxCount`, `charcType`, `hasFilter`, `isVariable`, `existNamedField`;

- для остальных характеристик UI явно показывает, что это не полный список WB, а `подсказки из карточек`; свое значение разрешено, если характеристика не помечена как строгий справочник;
- важное ограничение WB API: официальный `/content/v2/object/charcs/{subjectId}` не отдает полный список допустимых значений для произвольной характеристики вроде `Фактура материала` или длины; полный список ЛК WB для таких полей публичным API не найден;
- принято решение вернуться к расширенному словарю значений после получения MPStats API: MPStats/внешние данные могут стать источником подсказок по категории, а ручной ввод специалистов нужно будет накапливать в своем словаре `subjectID + charcID`.
- публичные фильтры WB-каталога обсуждались как возможный обходной источник значений, но не выбраны как основа: это не seller API, может быть нестабильно и требовать отдельного маппинга фильтр -> `charcID`; предпочтительный путь — MPStats API.
- список кабинетов и WB-эндпоинты усилены проверкой доступа: пользователь видит только свои/доступные порталы, глобальный доступ есть у admin/all/руководителя, `/api/wb/cards` и `/api/wb/characteristics` проверяют доступ к `portal_id` до получения токена;
- WB token meta пересчитывается от реального `iat` JWT-токена: срок считается от даты выпуска + 180 дней, старые сохраненные токены декодируются на backend из encrypted storage без отдачи токена во frontend.
- экран списка карточек развит в рабочую поверхность: поиск по карточкам, фильтры по проблемности/статусу/категории, явные причины проверки, счетчики и ограниченный рабочий набор карточек с сохранением выбора в `localStorage` по кабинету.

Что осталось:
- решить, нужно ли переносить ограниченный рабочий набор карточек из `localStorage` в backend, чтобы выбор был общим для команды и разных браузеров;
- проверить качество MPStats-подсказок характеристик на реальных карточках и при необходимости добавить маппинг названий WB `charcID` <-> MPStats-отчет;
- после MPStats решить модель словаря значений: `subjectID + charcID + source` (`wb_directory`, `mpstats`, `portal_cards`, `manual`);
- сверить XLSX-выгрузку контента с реальным категорийным шаблоном WB из ЛК, потому что WB формирует точные контент-шаблоны под предмет/категорию.

- `AGENTS.md` в репозитории отсутствует, триггеры doc-update выполняем вручную по измененным зонам (`auth`, `REST`, `frontend`, `маршруты`);

- env fallback `WB_API_TOKEN` разрешен только для `demo-wb`; числовые порталы должны использовать encrypted token через `set-wb-token`, чтобы не смешивать кабинеты.
